# 库存上报文档

> 给青龙脚本 / 自写采集脚本用。以后写库存上报代码，**只看这份文档**即可。  
> 对应接口：`POST /api/v1/stock-report`  
> 参考实现：`scripts/api_template.py` 里的 `StockReporter`

---

## 1. 接口概览

| 项 | 值 |
|---|---|
| Method | `POST` |
| Path | `/api/v1/stock-report` |
| 完整 URL 示例 | `http://127.0.0.1:5711/api/v1/stock-report` |
| Content-Type | `application/json` |
| 鉴权 | `Authorization: Bearer <API_TOKEN>` |
| 作用域 | **一次请求 = 一个小程序（program）的全量在架商品快照** |

`API_TOKEN` 与 `.env` 里的 `API_TOKEN` 一致；也可用环境变量：

```bash
POINTS_MONITOR_URL=http://127.0.0.1:5711
POINTS_MONITOR_TOKEN=你的token
```

---

## 2. 请求体结构

```json
{
  "program_id": "wxba70fb8e3eb3aab9",
  "products": [
    {
      "product_id": "sku_001",
      "product_name": "纯积分商品",
      "stock": 20,
      "points": 300,
      "image_url": "https://example.com/a.jpg"
    },
    {
      "product_id": "sku_002",
      "product_name": "积分加钱购礼盒",
      "stock": 5,
      "points": 800,
      "cash": 19.9,
      "image_url": "https://example.com/b.jpg"
    }
  ]
}
```

### 2.1 顶层字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `program_id` | string | 是 | 小程序唯一 ID（建议用 appid / 脚本内固定标识） |
| `products` | array | 是 | 当前在架商品列表；**建议报全量**，漏掉的会被系统判定为下架 |

### 2.2 `products[]` 字段

| 字段 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `product_name` | string | **是** | — | 商品名称 |
| `product_id` | string | 否 | 用 `product_name` 顶替 | 商品稳定 ID。**强烈建议传**，否则改名会当成新商品 |
| `stock` | int | 否 | `0` | 当前库存数量 |
| `points` | int | 否 | `0` | 兑换所需积分。纯积分商品只传这个即可 |
| `cash` | float | 否 | `0` | 积分加钱购所需现金，**单位：元**。`0` / 省略 = 纯积分商品 |
| `image_url` | string | 否 | — | 商品图片 URL（外链即可，系统会原样存） |
| `last_updated` | datetime | 否 | 服务端当前时间 | 可选时间戳；一般不用传 |

### 2.3 价格组合怎么传

| 商品类型 | `points` | `cash` | 示例 |
|---|---|---|---|
| 纯积分 | `> 0` | 省略或 `0` | `{"points": 500}` |
| 积分加钱购 | `> 0` | `> 0` | `{"points": 800, "cash": 19.9}` |
| 纯现金（极少见） | `0` / 省略 | `> 0` | `{"cash": 9.9}` |

> **不要在脚本里过滤「积分加钱购」商品。**  
> 系统已支持 `points + cash` 一起存、一起展示、一起筛选。原样上报即可。

数值兼容：

- `points` / `cash` 支持数字字符串，如 `"800"`、`"19.9"`
- `cash` 传 `null` / `""` 会按 `0` 处理
- `points` 会截成整数（`"80.9"` → `80`）

---

## 3. 服务端会做什么

1. **程序不存在则自动创建**（`program_name` 暂为 `Unknown Program`，后续可在前端改）
2. **按 `(program_id, product_id)` upsert 商品**
   - 新商品：写入 name / image / points / cash / stock
   - 老商品：刷新 name、image（有传才改）、**points、cash 每次都覆盖**、stock
3. **库存变化写 `stock_history`**
4. **下架检测（重要）**
   - 把「上一次有报告的那天」出现过、但本次没出现的商品，标成 `is_unlisted=1`（系统下架）
   - 再次出现在报告里会自动恢复上架
   - **不碰**用户手动隐藏（`is_hidden`）
5. 历史日志按系统设置自动裁剪

因此脚本侧应尽量：

- **每次上报该小程序当前全部在架商品**（全量快照，不是增量）
- `product_id` 稳定不变
- 没货就报 `stock: 0`，不要直接从列表删掉（删掉会被当成下架）

---

## 4. 响应

成功（HTTP 200）：

```json
{
  "status": "success",
  "updated_products": ["sku_001", "sku_002"],
  "unlisted_products": ["sku_old_gone"]
}
```

| 字段 | 说明 |
|---|---|
| `updated_products` | 本次新建，或库存发生变化的 `product_id` 列表 |
| `unlisted_products` | 本次被系统标记为下架的 `product_id` 列表 |

常见失败：

| HTTP | 原因 |
|---|---|
| 401 / 403 | Token 错误或缺失 |
| 422 | 请求体校验失败（缺 `program_id` / `product_name` 等） |
| 5xx | 服务端异常 |

## 5. 最小可跑示例

```python
# -*- coding: utf-8 -*-
import os
import requests

API_URL = os.getenv("POINTS_MONITOR_URL", "http://127.0.0.1:5711")
API_TOKEN = os.getenv("POINTS_MONITOR_TOKEN", "YOUR_TOKEN_HERE")

def report_stock(program_id, products):
    url = f"{API_URL.rstrip('/')}/api/v1/stock-report"
    headers = {
        "Authorization": f"Bearer {API_TOKEN}",
        "Content-Type": "application/json",
    }
    payload = {
        "program_id": program_id,
        "products": products,
    }
    resp = requests.post(url, json=payload, headers=headers, timeout=15)
    print(resp.status_code, resp.text)
    resp.raise_for_status()
    return resp.json()


# 从业务接口拉到的商品，原样映射即可（无需过滤积分加钱购）
raw_items = [
    {"id": "a1", "name": "纯积分券", "stock": 20, "points": 300, "cash": 0, "img": "..."},
    {"id": "b2", "name": "加钱购礼盒", "stock": 5, "points": 800, "cash": 19.9, "img": "..."},
]

products = []
for item in raw_items:
    products.append({
        "product_id": str(item["id"]),
        "product_name": item["name"],
        "stock": int(item.get("stock") or 0),
        "points": int(item.get("points") or 0),
        # 有现金就传；没有 / 0 表示纯积分
        "cash": float(item.get("cash") or 0),
        "image_url": item.get("img") or None,
    })

report_stock("your_program_id", products)
```

也可以直接复用仓库模板：


## 7. 前端怎么用这些数据（写脚本时心里有数）

- 库存管理页展示价格：
  - `cash == 0` → `300 积分`
  - `cash > 0` → `800 积分 + ¥19.9`
- 价格筛选：
  - 纯积分
  - 积分加钱购（可再限现金上限，如 ≤ ¥1）
  - 全部
- 可兑换判断：
  - 需要 `用户最高积分 >= points`
  - 若 `cash > 0`，还需要 `用户最高现金 >= cash`

用户积分 / 现金来自另一条接口：`POST /api/v1/qinglong/report`（账号维度的余额上报，不是库存）。

## 8. 脚本编写 checklist

- [ ] 带了 `Authorization: Bearer <token>`
- [ ] 一个小程序一次请求，`program_id` 固定
- [ ] `products` 是**当前在架全量**，不是增量补丁
- [ ] 每个商品有稳定 `product_id`
- [ ] 必填 `product_name`
- [ ] 纯积分：只传 `points`（或 `cash: 0`）
- [ ] 积分加钱购：同时传 `points` + `cash`（元）
- [ ] **不要过滤掉加钱购商品**
- [ ] 没货用 `stock: 0`，不要从列表删除（除非确认已下架）
- [ ] 超时建议 ≥ 15s（商品多时）

## 10. 和「积分上报」的区别（别混）

| | 库存上报 | 积分/现金余额上报 |
|---|---|---|
| 接口 | `POST /api/v1/stock-report` | `POST /api/v1/qinglong/report` |
| 主体 | 小程序商品 | 微信号 / 手机号账号 |
| 关键字段 | `points` + `cash`（**兑换价**） | `current_points` + `current_cash`（**账户余额**） |
| 是否全量 | 建议全量商品快照 | 报该账号涉及的小程序即可 |

两套数据独立：库存告诉你「货多少、兑多少钱」；积分上报告诉你「账号手里有多少」。
